🔍 เก็บ & ขุด requirement

ถามให้ครบก่อนตีราคา — ไม่เดาแทนลูกค้า:

เป้าหมายธุรกิจผู้ใช้ / บทบาท ขอบเขต in/outปริมาณงาน (scale) การเชื่อมต่อระบบnon-functional ข้อจำกัด งบ/เวลา/เทคโนโลยี
  • ข้อไหนยังไม่ได้คำตอบ → บันทึกเป็น assumptions[]

  • แตกฟีเจอร์เป็น modules[] จัดกลุ่ม A/B/C/D (groupLegend) — นี่คือ ฐานของการตีราคา
  • actors[] · scope.in/out · useCases[] · nonFunctional[] · risks[]

💰 ตีราคา (man-day rate, THB)

  • ประเมิน man-day low–high ต่อ module ตาม complexity
  • ใส่ support 4 ตัวเสมอ วิเคราะห์/ออกแบบQADeployPM
  • เรต default หลายระดับ + เผื่อความเสี่ยง (contingencyPct ~20%)
ดูเรต + สูตรราคา + โหมด AI
เรตบาท/วัน
Junior2,500
Mid4,000
Senior6,000
Expert8,000

subtotal = Σ(modules.md) + Σ(support.md) → total = round(subtotal × (1 + contingencyPct/100)) → price = man-day × เรต

โหมด AI (ถ้าทำ · internal): AI md = ceil(คน ÷ factor) — dev 3–5x · support 1.3–2.5x · contingency 25% · เรตเท่าเดิม (ราคาลงเพราะวันน้อยลง) · ทุก aiMd มี factor กำกับ

📅

🗓️ ร่างแผนงาน (schedule)

~4–6 เฟส แบ่งตามกลุ่มโมดูล (A/B/C/D) + เฟสปิดท้าย QA/UAT/Deploy

  • วัน = relative working day (วันที่ 1 = วันเริ่มจริง · จ–ศ · ไม่ผูกปฏิทิน)
  • ช่วงวันรวม ≈ man-day ÷ ขนาดทีม · render เป็น Gantt + ไทม์ไลน์แนวตั้ง ใน §7
  • ⚠️ กำหนดการของ โปรเจกต์ลูกค้า — ไม่ใช่ timeline ของ AI agent
4

📝 เขียนไฟล์ data/<uuid>/data.json

ประกอบทุกอย่างเป็นไฟล์เดียว ตาม contract (type Project):

{
  "projectName": "ระบบคูปอง (Coupon System)",
  "summary": { priceLow, priceHigh, manDayLow, manDayHigh },
  "modules": [ { group, code, name, complexity, mdLow, mdHigh } ],
  "support": [...], "rates": [...], "contingencyPct": 20,
  "schedule": { phases:[ {name,startDay,endDay,items} ] },
  "ai": { ... } // optional — internal เท่านั้น
}
